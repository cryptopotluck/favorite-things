Software Engineer Hiring Test

  • A few technical questions

To submit your results, create a GitHub or GitLab repo called favorite-things. This is what you enter into the job submission fields.

Technical Questions

  1. Please describe yourself using JSON in a file called myself.json.

Please answer the following questions in a markdown file called answers.md.

  1. How long did you spend on the coding test below? What would you add to your solution if you had more time? If you didn't spend much time on the coding test then use this as an opportunity to explain what you would add.
  2. What was the most useful feature that was added to the latest version of your chosen language? Please include a snippet of code that shows how you've used it.

Coding Test

This fun coding test allows us to evaluate your ability to understand requirements and your coding style and habits. Your time is valuable so we encourage you to find ways to maximize results for minimal effort. The project is an application that allows the user to track their favorite things. The necessary data to track is:

  • Title: Required field. A short, descriptive name of a favorite thing.
  • Description: Optional field. A long description of a favorite thing. Can be empty, but if it has any text then then minimum length is 10 characters.
  • Ranking: Required numeric field. A numbered priority according to the user (1...n). Ranking numbers should not repeat for the given category, so if a ranking is set on a new favorite thing as "1", then all other favorite things for that favorite should be reordered.
  • Metadata: Optional field. A key/value store where valid data types include text, number, date, and enum types.
  • Category: Required data per favorite thing. Selectable between person, place, or food, but users must be able add more categories.
  • Created Date: Required automatic field. Not entered by user but visible to them.
  • Modified Date: Required automatic field. Not entered by user but visible to them.
  • Audit Log: Automatically generated on data. All changes to data must be logged. Not entered by user but visible to them.

Backend

For the backend, this is your chance to show us that you know how to create clean REST or GraphQL APIs. You must support all of the data specified above. Represent all the data with at least the following:

API Type Representation
REST GET, PUT, POST
GraphQL Queries, Mutations

The backend must be built using these options:

  • OS: Any flavor of Linux
  • Server Side Scripting: Python 2.7+ or 3.6+
  • Server Framework: Django or Flask
  • Backend Database: PostgreSQL (Django), MySQL (Django or Flask), DynamoDB (Flask)
  • ORM: Django or SQLAlchemy (Flask-only) or Boto3 (DynamoDB only)
  • Deployment: Any means you prefer

Frontend

The frontend is all about how the user interacts with the data. All data must be visible to the user, including a list of favorite things within a category.

Coding Test Deliverables

    • Bonus points if you include a .gitignore file.
    • Bonus points if you demonstrate steps taken to autolint your code.

  2. An entity relationship diagram, which depicts the table(s) and their relationship to one another.

  3. A well-tested REST or GraphQL API written in Python.

    • If using Django, you must use Django REST Framework's Class-Based Views. Projects with Function-based views will be automatically rejected.
    • If using Flask, you must return JSON responses.

  4. A deployed version of your project

  • Bonus points for an attractive yet usable interface. Use any Javascript framework but understand that at BriteCore we are moving from JQuery/Knockout.js to VueJS
  • Mega Bonus points for using VueJS
  • Mega bonus points squared for using VueJS as a Single Page App
